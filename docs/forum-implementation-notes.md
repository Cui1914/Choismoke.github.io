# 烟灰缸论坛实现判断

## 1. 结论

`烟灰缸` 不适合继续只用纯静态 HTML 直接做完，因为：

- 需要邮箱注册登录
- 需要账号体系与会话
- 需要发帖、评论、私信、通知
- 需要图片 / 视频上传
- 需要后台管理、举报处理、封禁

这些都要求服务端、数据库和对象存储。GitHub 仓库仍然可以继续作为代码仓库，但不能再只依赖 GitHub Pages 作为完整承载方案。

## 2. 推荐架构

- 前端：`Next.js`
- 数据库：`PostgreSQL`
- 认证：邮箱登录 + 会话管理
- 文件存储：对象存储（图片 / 视频）
- 后台：与主站共仓库，管理端作为受保护路由
- 部署：
  - 代码继续放当前 GitHub 仓库
  - 前端和服务端部署到支持 Node.js 的平台
  - 静态主站可继续保留，也可逐步并入同一应用

## 3. 渐进式路线

### 阶段 1：当前仓库先做前端骨架

- 新增论坛首页
- 新增论坛列表、帖子详情、登录注册、个人主页的静态页面
- 确定统一视觉语言和信息架构

### 阶段 2：迁移为动态应用

- 引入应用框架
- 设计数据库表
- 落地注册登录
- 落地帖子、评论、点赞、收藏、私信、通知、后台

### 阶段 3：正式上线

- 从 GitHub Pages 迁移论坛子系统
- 保留主站内容并接入统一导航
- 使用同一账号体系

## 4. 当前建议

- 先在当前仓库里把 `烟灰缸` 的前端雏形做完整
- 然后再切换到动态栈做真实论坛

这样不会推翻现有主站，也能让设计和产品结构先稳定下来
